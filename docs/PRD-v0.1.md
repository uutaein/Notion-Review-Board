> 이 문서는 Notion에 축적된 학습 문서를 문서 단위로 복습하기 위한 Electron 기반 학습 관리 앱의 PRD다.  
> 제품의 핵심은 여러 Notion 데이터베이스에서 개별 페이지를 수집하고, 이를 하나의 통합 복습 큐로 병합한 뒤, FSRS 기반 간격 반복과 칸반/목록 워크플로우로 복습을 처리하는 것이다.

## 0. 변경 이력

| 버전 | 변경 내용 |
| --- | --- |
| v0.1 | Notion Review Board 기본 PRD 초안 작성. 복습 단위, 여러 Notion DB 등록, 수집 기준 3종, 필드 매핑, 4단계 평가, 삭제/변경 페이지 확인 정책 반영 |

### 0.1 현재 확정 기준

2026-06-11 기준 현재 제품 방향은 다음과 같다.

1. 앱은 Electron 기반 데스크톱 앱으로 만든다.
2. iPad 네이티브 앱은 1차 범위에 포함하지 않는다.
3. iPad 사용은 원격 데스크톱으로 PC의 Electron 앱을 조작하는 방식까지 고려한다.
4. 복습 단위는 Notion 데이터베이스 자체가 아니라 데이터베이스 안의 개별 Notion 페이지다.
5. 사용자는 복습 대상으로 사용할 Notion 데이터베이스를 1개 이상 등록할 수 있다.
6. 여러 Notion 데이터베이스에서 수집된 페이지는 하나의 통합 Review Queue로 병합한다.
7. 앱은 Notion 페이지 본문 전체를 저장하지 않고, Notion Page ID, URL, 제목, 태그/분류 메타데이터, FSRS 상태, 복습 기록을 저장한다.
8. 사용자는 각 Notion 데이터베이스별로 제목, URL, 분류, 태그, 체크박스 등 필요한 속성을 직접 매핑할 수 있어야 한다.
9. 복습 대상 수집 기준은 태그/분류 기반, 체크박스 기반, 데이터베이스 전체 수집을 모두 지원한다.
10. 복습 평가는 다시, 어려움, 보통, 쉬움의 4단계로 제공한다.
11. 4단계 평가는 FSRS의 Again, Hard, Good, Easy에 매핑한다.
12. 오늘 복습 목록은 날짜순, 랜덤순, 특정 태그순으로 볼 수 있어야 한다.
13. 태그 또는 분류 값이 없는 페이지는 "미분류"로 표시한다.
14. 삭제된 페이지와 변경된 페이지는 별도 화면에서 확인할 수 있어야 한다.
15. 변경된 페이지는 자동으로 일정을 바꾸지 않고, 사용자가 "오늘 복습으로 당기기" 또는 "기존 일정 유지"를 선택한다.

## 1. 배경

사용자는 Notion을 학습 문서 저장소로 사용한다.  
Notion에는 시험, 개발, AI, 네트워크, 보안, 프로젝트 관리 등 다양한 학습 문서가 데이터베이스 형태로 쌓인다.

하지만 현재 방식에는 다음 문제가 있다.

1. Notion에 학습 자료는 계속 쌓이지만 주기적으로 다시 보기가 어렵다.

   * 문서를 작성한 뒤 시간이 지나면 다시 볼 시점을 놓친다.
   * 중요한 문서와 덜 중요한 문서가 섞여 복습 우선순위를 정하기 어렵다.
   * 여러 Notion 데이터베이스에 학습 자료가 흩어져 있으면 오늘 무엇을 복습해야 하는지 알기 어렵다.

2. Anki는 카드 단위 복습에는 강하지만 문서 단위 복습에는 불편하다.

   * Notion 문서 하나를 그대로 복습 대상으로 다루기 어렵다.
   * 문서 URL을 열어 읽고, 이해도를 기준으로 다음 복습일을 조정하는 흐름이 자연스럽지 않다.
   * 카드 생성 없이 문서 자체를 복습하고 싶은 사용 흐름과 맞지 않는다.

3. Notion 데이터베이스마다 구조가 다를 수 있다.

   * 어떤 데이터베이스는 `분류` 속성이 있다.
   * 어떤 데이터베이스는 `암기` 체크박스가 있다.
   * 어떤 데이터베이스는 전체 페이지를 모두 복습 대상으로 삼고 싶을 수 있다.
   * 제목, 태그, 출처, 분류 속성명이 데이터베이스마다 다를 수 있다.

4. 여러 데이터베이스를 섞어 복습하기 어렵다.

   * AI, 네트워크, 보안, 소프트웨어, 프로젝트 관리 등 도메인별 DB가 나뉘어 있을 수 있다.
   * 사용자는 여러 DB에서 오늘 볼 항목을 하나의 목록으로 보고 싶다.
   * 날짜순, 랜덤순, 특정 태그순으로 복습 목록을 바꿔가며 보고 싶다.

5. 문서 변경과 삭제 상태를 복습 큐와 연결하기 어렵다.

   * Notion 페이지가 변경되면 다시 봐야 할 수 있다.
   * Notion 페이지가 삭제되면 앱 안의 복습 항목도 더 이상 열 수 없다.
   * 단순히 동기화 목록에서 사라졌다고 즉시 삭제하면 권한 오류나 일시적 동기화 실패와 구분하기 어렵다.

## 2. 목표

이 제품의 목표는 **Notion을 지식 저장소로 유지하면서, Electron 앱이 문서 단위 복습 큐와 FSRS 기반 다음 복습일을 관리하는 것**이다.

### 2.1 핵심 목표

1. 사용자가 지정한 여러 Notion 데이터베이스에서 복습 대상 페이지를 수집한다.
2. Notion 데이터베이스 안의 개별 페이지를 복습 단위로 관리한다.
3. 여러 데이터베이스에서 수집한 페이지를 하나의 통합 Review Queue로 병합한다.
4. 사용자가 데이터베이스별로 제목, URL, 태그, 분류, 체크박스 속성을 매핑할 수 있게 한다.
5. 태그/분류 기반, 체크박스 기반, 데이터베이스 전체 수집을 모두 지원한다.
6. 오늘 복습해야 할 문서를 `dueAt <= today` 기준으로 보여준다.
7. 사용자가 Electron 내부 뷰어에서 Notion 페이지 URL을 열어 직접 읽을 수 있게 한다.
8. 복습 후 다시, 어려움, 보통, 쉬움 4단계로 평가할 수 있게 한다.
9. 평가 결과에 따라 FSRS 기반으로 다음 복습일을 계산한다.
10. 날짜순, 랜덤순, 특정 태그순으로 복습 목록을 볼 수 있게 한다.
11. 태그가 없는 페이지는 "미분류"로 표시한다.
12. 삭제된 페이지와 변경된 페이지를 별도 화면에서 확인할 수 있게 한다.
13. 변경된 페이지는 사용자가 오늘 복습으로 당기거나 기존 일정을 유지할 수 있게 한다.
14. 복습 기록을 저장해 통계, heatmap, 누적 복습 수의 기반을 만든다.

### 2.2 제품이 해결하려는 핵심 문제

> 현재 문제의 본질은 “학습 자료가 부족한 것”이 아니라, **Notion에 쌓인 문서를 다시 볼 시점과 우선순위를 관리하는 시스템이 없다는 것**이다.

따라서 1차 제품은 복잡한 퀴즈 생성보다 다음에 집중한다.

1. 어떤 Notion 페이지를 복습 대상으로 삼을 것인가?
2. 여러 데이터베이스에서 가져온 페이지를 어떻게 하나의 큐로 합칠 것인가?
3. 오늘 다시 봐야 할 문서는 무엇인가?
4. 문서를 읽고 난 뒤 얼마나 잘 기억했는가?
5. 다음 복습일은 언제인가?
6. 삭제되거나 변경된 Notion 페이지를 어떻게 확인할 것인가?

## 3. Non-Goals

이번 1차 범위에서는 아래 기능을 포함하지 않는다.

1. iPad 네이티브 앱 개발
2. 모바일 앱 개발
3. 웹 SaaS 서비스 운영
4. 여러 사용자 계정과 팀 단위 공유
5. Notion 페이지 본문 전체 저장
6. Notion 페이지 본문을 앱 DB에 복제하는 기능
7. AI 기반 자동 문제 생성
8. AI 기반 요약 생성
9. Anki 카드 자동 생성
10. Anki 데크 양방향 동기화
11. Notion 문서 편집 기능
12. Notion 데이터베이스 구조 자동 변경
13. Notion 권한 관리 자동화
14. 복잡한 사용자 인증 시스템
15. 클라우드 서버 기반 중앙 동기화
16. 실시간 협업 기능
17. 모바일 푸시 알림
18. 캘린더 연동
19. 이메일 알림
20. 고급 통계 리포트
21. 자동 백업/클라우드 백업
22. OCR 또는 PDF 본문 추출
23. 문서 내용 기반 검색 인덱싱
24. 다중 기기 간 실시간 동기화

> 1차 목표는 대형 학습 플랫폼이 아니라, 개인 사용자가 Notion 문서를 문서 단위로 다시 보게 만드는 **작고 실용적인 Electron 기반 복습 큐 앱**이다.

## 4. 복습 대상과 용어

### 4.1 Review Source

Review Source는 사용자가 복습 대상으로 등록한 Notion 데이터베이스를 의미한다.

사용자는 Review Source를 1개 이상 등록할 수 있다.

각 Review Source는 다음 정보를 가진다.

| 항목 | 설명 |
| --- | --- |
| 소스 이름 | 앱 안에서 표시할 데이터베이스 이름 |
| Notion Database ID 또는 URL | 수집 대상 Notion 데이터베이스 식별 정보 |
| 활성 여부 | 동기화 대상 포함 여부 |
| 수집 방식 | 태그/분류 기반, 체크박스 기반, 전체 수집 |
| 필드 매핑 | 제목, URL, 분류, 태그, 체크박스 등 속성 매핑 |
| 마지막 동기화 시각 | 해당 소스의 마지막 동기화 기준 |

### 4.2 Review Item

Review Item은 실제 복습 대상이다.

본 제품의 Review Item은 Notion 데이터베이스 안의 개별 페이지 1개를 의미한다.

Review Item은 최소한 다음 정보를 가진다.

| 항목 | 설명 |
| --- | --- |
| Notion Page ID | 중복 판단과 동기화의 1차 기준 |
| URL | Electron 내부 뷰어 또는 외부 브라우저에서 열 Notion 페이지 URL |
| 제목 | 복습 목록에 표시할 이름 |
| 원본 Source | 어떤 Review Source에서 수집되었는지 |
| 분류/태그 | 필터링과 그룹화에 사용할 메타데이터 |
| dueAt | 다음 복습 예정일 |
| lastReviewedAt | 마지막 복습 시각 |
| fsrsState | FSRS 계산에 필요한 상태 |
| status | active, changed, deleted, sync_error 등 상태 |
| createdAt | 앱에 처음 등록된 시각 |
| updatedAt | 앱에서 마지막으로 갱신된 시각 |

### 4.3 Review Log

Review Log는 사용자가 복습을 수행한 기록이다.

각 복습 완료 시 다음 정보가 저장된다.

| 항목 | 설명 |
| --- | --- |
| Review Item | 어떤 페이지를 복습했는지 |
| 평가 | 다시, 어려움, 보통, 쉬움 |
| reviewedAt | 복습 수행 시각 |
| previousDueAt | 복습 전 예정일 |
| nextDueAt | 복습 후 계산된 다음 예정일 |
| source | 당시 원본 Review Source |
| category | 당시 분류/태그 |

## 6. 핵심 사용자 시나리오

### 6.1 Notion 데이터베이스 등록

사용자로서, 복습 대상으로 사용할 Notion 데이터베이스를 앱에 등록하고 싶다.  
그래야 Notion에 쌓인 학습 문서를 앱이 수집할 수 있다.

### 6.2 데이터베이스별 필드 매핑

사용자로서, 각 Notion 데이터베이스마다 제목, 분류, 태그, 체크박스 속성을 직접 매핑하고 싶다.  
그래야 데이터베이스마다 속성명이 달라도 복습 큐에 정확히 들어올 수 있다.

### 6.3 태그/분류 기반 수집

사용자로서, 특정 도메인 태그가 있는 페이지 중 일부만 복습 대상으로 가져오고 싶다.  
예를 들어 `분류 = AI`인 페이지를 복습 큐에 넣고 싶다.

### 6.4 체크박스 기반 수집

사용자로서, 내가 앱에서 지정한 Notion 체크박스가 체크된 페이지만 복습 대상으로 가져오고 싶다.  
그래야 Notion에서 직접 복습 여부를 표시할 수 있다.

### 6.5 데이터베이스 전체 수집

사용자로서, 특정 Notion 데이터베이스 안의 모든 페이지를 복습 큐에 넣고 싶다.  
그래야 별도 태그나 체크박스를 붙이지 않아도 전체 문서를 카드처럼 복습할 수 있다.

### 6.6 오늘 복습 목록 확인

사용자로서, 오늘 복습해야 할 Notion 페이지 목록을 보고 싶다.  
그래야 어떤 문서를 다시 봐야 하는지 바로 알 수 있다.

### 6.7 Notion 페이지 열람

사용자로서, 복습 목록에서 문서를 선택하면 Electron 앱 안에서 Notion 페이지를 열고 싶다.  
그래야 앱을 벗어나지 않고 문서를 읽을 수 있다.

### 6.8 난이도 평가

사용자로서, 문서를 읽은 뒤 다시, 어려움, 보통, 쉬움 중 하나로 평가하고 싶다.  
그래야 기억 정도에 따라 다음 복습일이 달라질 수 있다.

### 6.9 목록 정렬과 섞기

사용자로서, 오늘 복습 목록을 날짜순, 랜덤순, 특정 태그순으로 바꿔 보고 싶다.  
그래야 복습 흐름을 상황에 맞게 조절할 수 있다.

### 6.10 미분류 페이지 처리

사용자로서, 태그가 없는 페이지도 복습 목록에서 누락되지 않았으면 좋겠다.  
그래야 아직 분류하지 않은 문서도 복습할 수 있다.

### 6.11 변경된 페이지 확인

사용자로서, 마지막 동기화 이후 변경된 Notion 페이지를 별도 화면에서 보고 싶다.  
그래야 내용이 바뀐 문서를 오늘 다시 볼지 결정할 수 있다.

### 6.12 삭제된 페이지 확인

사용자로서, Notion에서 삭제되어 더 이상 열 수 없는 페이지를 별도 화면에서 확인하고 싶다.  
그래야 복습 큐를 정리할 수 있다.

## 7. 핵심 기능

### 7.1 Notion 연동 설정

사용자는 앱에서 Notion 연동 정보를 설정할 수 있어야 한다.

필요한 설정은 다음과 같다.

1. Notion API 토큰
2. 복습 대상으로 사용할 Notion 데이터베이스 URL 또는 ID
3. 데이터베이스별 표시 이름
4. 데이터베이스별 활성/비활성 여부
5. 데이터베이스별 수집 방식
6. 데이터베이스별 필드 매핑

MVP에서는 사용자가 직접 Notion API 토큰을 입력하고, Notion 데이터베이스 URL 또는 ID를 등록하는 방식을 우선한다.

### 7.2 Review Source 등록

사용자는 복습 소스를 1개 이상 등록할 수 있어야 한다.

Review Source 등록 시 필요한 정보는 다음과 같다.

| 항목 | 필수 여부 | 설명 |
| --- | --- | --- |
| 소스 이름 | 필수 | 앱에서 표시할 이름 |
| Notion Database URL 또는 ID | 필수 | 복습 대상으로 사용할 Notion 데이터베이스 |
| 수집 방식 | 필수 | 태그/분류 기반, 체크박스 기반, 전체 수집 |
| 제목 속성 매핑 | 필수 | 페이지 제목으로 사용할 Notion 속성 |
| URL 속성 매핑 | 선택 | 별도 URL 속성이 없으면 Notion Page URL 사용 |
| 분류/태그 속성 매핑 | 선택 | 필터와 그룹화에 사용할 속성 |
| 체크박스 속성 매핑 | 조건부 필수 | 체크박스 기반 수집 시 필수 |
| 활성 여부 | 필수 | 동기화 대상 포함 여부 |

### 7.3 필드 매핑

앱은 Notion 데이터베이스마다 다른 속성명을 처리하기 위해 필드 매핑 기능을 제공해야 한다.

매핑 대상은 다음과 같다.

1. 제목 속성
2. URL 속성
3. 분류 속성
4. 태그 속성
5. 체크박스 속성
6. 출처 속성
7. updated_at 또는 마지막 수정 시각

MVP에서는 모든 속성 자동 추론을 목표로 하지 않는다.  
사용자가 데이터베이스별로 필요한 속성을 직접 선택하는 방식을 우선한다.

### 7.4 복습 대상 수집 기준

앱은 Review Source별로 다음 수집 기준을 지원한다.

#### 7.4.1 태그/분류 기반 수집

사용자는 특정 속성에 특정 값이 있는 페이지만 복습 대상으로 수집할 수 있다.

예시는 다음과 같다.

```text
분류 = AI
분류 = NW
도메인 = Backend
태그 contains review
```

#### 7.4.2 체크박스 기반 수집

사용자는 특정 체크박스 속성이 체크된 페이지만 복습 대상으로 수집할 수 있다.

예시는 다음과 같다.

```text
암기 = checked
복습대상 = checked
Review = checked
```

#### 7.4.3 데이터베이스 전체 수집

사용자는 특정 데이터베이스의 모든 페이지를 복습 대상으로 수집할 수 있다.

이 방식에서는 태그나 체크박스가 없어도 모든 페이지가 Review Queue에 추가된다.

### 7.5 통합 Review Queue

앱은 여러 Review Source에서 수집한 Review Item을 하나의 통합 Review Queue로 병합한다.

Queue 병합 규칙은 다음과 같다.

1. Review Item의 1차 식별 기준은 Notion Page ID다.
2. 동일한 Notion Page ID가 다시 수집되면 중복 생성하지 않고 기존 항목을 갱신한다.
3. 같은 페이지가 여러 Review Source에서 발견될 수 있는 경우, 원본 source 정보를 보존한다.
4. Notion Page ID가 다르지만 URL이 같은 항목은 자동 병합하지 않고 중복 후보로 볼 수 있다.
5. 제목만 같은 항목은 중복으로 보지 않는다.

### 7.6 오늘 복습 목록

오늘 복습 목록은 다음 조건을 만족하는 Review Item으로 생성한다.

```sql
due_at <= today
status = 'active'
```

오늘 복습 목록은 다음 정보를 표시한다.

1. 제목
2. 원본 Review Source
3. 분류 또는 태그
4. 출처
5. dueAt
6. lastReviewedAt
7. 복습 상태
8. 문서 열기 버튼
9. 난이도 평가 버튼

### 7.7 목록 보기 방식

오늘 복습 목록은 다음 보기 방식을 제공한다.

| 보기 방식 | 설명 |
| --- | --- |
| 날짜순 | dueAt 오름차순으로 표시 |
| 랜덤순 | 오늘 복습 대상 안에서 무작위 순서로 표시 |
| 특정 태그순 | 사용자가 선택한 태그 또는 분류 기준으로 필터링하거나 그룹화 |
| 미분류 | 태그 또는 분류 값이 없는 항목을 "미분류"로 표시 |

MVP에서는 날짜순을 기본값으로 한다.  
랜덤순은 사용자가 버튼을 눌러 오늘 목록 순서를 섞는 방식으로 제공한다.  
특정 태그순은 분류/태그 속성이 매핑된 Review Source에서 우선 지원한다.

### 7.8 문서 뷰어

사용자는 Review Item을 선택해 Notion 페이지를 열람할 수 있어야 한다.

MVP 기본 동작은 다음과 같다.

1. Review Item의 URL을 Electron 내부 뷰어에서 연다.
2. Electron 내부 뷰어에서 열람이 실패하면 외부 브라우저로 열기 버튼을 제공한다.
3. 앱은 Notion 페이지 본문 전체를 저장하지 않는다.
4. 앱은 Notion Page ID와 URL을 기준으로 문서 열람을 처리한다.

문서 뷰어 화면은 다음 정보를 함께 제공해야 한다.

1. 현재 문서 제목
2. 원본 Review Source
3. 분류/태그
4. 마지막 복습일
5. 다음 복습 예정일
6. 난이도 평가 버튼

### 7.9 복습 평가

사용자는 문서를 읽은 뒤 4단계로 평가한다.

| 앱 평가 | FSRS 매핑 | 의미 |
| --- | --- | --- |
| 다시 | Again | 거의 기억나지 않거나 다시 봐야 함 |
| 어려움 | Hard | 이해했지만 기억 유지가 약함 |
| 보통 | Good | 적절히 기억하고 있음 |
| 쉬움 | Easy | 쉽게 기억하고 있음 |

평가 완료 시 앱은 다음 작업을 수행한다.

1. Review Log를 생성한다.
2. FSRS 상태를 갱신한다.
3. next dueAt을 계산한다.
4. Review Item의 lastReviewedAt을 갱신한다.
5. 해당 항목을 오늘 복습 목록에서 완료 처리한다.

### 7.10 칸반 보드

칸반 보드는 문서 복습 워크플로우를 시각적으로 처리하기 위한 화면이다.

MVP 또는 P1 범위에서 다음 컬럼을 제공한다.

```text
오늘 복습
다시
어려움
보통
쉬움
```

사용자는 카드를 각 컬럼으로 드래그해 평가할 수 있다.

다만 원격 데스크톱 또는 터치 환경에서는 드래그가 불편할 수 있으므로, 난이도 버튼 평가는 반드시 함께 제공한다.

### 7.11 삭제된 페이지 화면

앱은 동기화 과정에서 삭제된 것으로 확인된 페이지를 별도 화면에서 보여준다.

삭제된 페이지 화면은 다음 정보를 표시한다.

1. 제목
2. 원본 Review Source
3. Notion Page ID
4. URL
5. 마지막 정상 동기화 시각
6. 삭제 감지 시각
7. 현재 상태
8. 큐에서 제거 또는 기록 보존 처리 버튼

삭제 판단은 단순히 목록에서 누락된 것만으로 확정하지 않는다.  
MVP에서는 누락된 항목을 삭제 후보로 표시하고, Page ID 재확인 또는 후속 동기화 결과를 통해 deleted 상태로 전환한다.

### 7.12 변경된 페이지 화면

앱은 Notion의 수정 시각이 앱에 저장된 마지막 동기화 시각 이후로 변경된 페이지를 별도 화면에서 보여준다.

변경된 페이지 화면은 다음 정보를 표시한다.

1. 제목
2. 원본 Review Source
3. 분류/태그
4. 마지막 복습 시각
5. 마지막 동기화 시각
6. Notion 수정 시각
7. 현재 dueAt
8. 선택 가능한 처리

변경된 페이지에 대한 처리 방식은 다음과 같다.

1. 오늘 복습으로 당기기
2. 기존 일정 유지

MVP에서는 변경 감지 후 자동으로 dueAt을 바꾸지 않는다.  
사용자가 변경된 페이지 화면에서 직접 처리한다.

### 7.13 동기화

앱은 사용자가 수동으로 동기화를 실행할 수 있어야 한다.

MVP 동기화는 수동 동기화를 기본으로 한다.

동기화 시 앱은 다음 작업을 수행한다.

1. 활성화된 Review Source 목록을 확인한다.
2. 각 Source의 수집 기준과 필드 매핑을 읽는다.
3. Notion에서 조건에 맞는 페이지를 조회한다.
4. 새 페이지를 Review Queue에 추가한다.
5. 기존 페이지의 제목, URL, 태그, 분류, updated_at을 갱신한다.
6. Notion에서 변경된 페이지를 changed 상태 또는 변경 목록에 표시한다.
7. Notion에서 삭제된 것으로 보이는 페이지를 삭제 후보 또는 deleted 상태로 표시한다.
8. 동기화 오류가 발생한 항목은 sync_error로 표시한다.

자동 동기화는 후속 버전에서 지원한다.

## 9. 비즈니스 규칙

### 9.1 복습 단위

1. 복습 단위는 Notion 데이터베이스 안의 개별 페이지다.
2. Notion 데이터베이스는 복습 소스이며, 복습 항목 자체가 아니다.
3. Review Item의 1차 식별 기준은 Notion Page ID다.
4. 앱은 Notion 페이지 본문 전체를 저장하지 않는다.
5. 앱은 페이지 열람을 위해 Notion Page ID와 URL을 저장한다.

### 9.2 여러 데이터베이스 병합

1. 사용자는 여러 Notion 데이터베이스를 Review Source로 등록할 수 있다.
2. 앱은 활성화된 Review Source에서 복습 대상 페이지를 수집한다.
3. 여러 Source에서 수집된 페이지는 하나의 Review Queue로 병합한다.
4. 동일한 Notion Page ID는 중복 등록하지 않는다.
5. 동일 URL이지만 Page ID가 다른 경우 자동 병합하지 않는다.

### 9.3 필드 매핑

1. 각 Review Source는 별도의 필드 매핑을 가진다.
2. 제목 속성은 필수로 매핑해야 한다.
3. URL 속성은 선택이다.
4. URL 속성이 없으면 Notion Page URL을 사용한다.
5. 태그/분류 속성은 선택이다.
6. 태그/분류 값이 없으면 "미분류"로 표시한다.
7. 체크박스 기반 수집을 사용할 경우 체크박스 속성 매핑은 필수다.

### 9.4 수집 기준

1. 태그/분류 기반 수집은 지정한 속성과 값 조건에 맞는 페이지만 수집한다.
2. 체크박스 기반 수집은 지정한 체크박스가 체크된 페이지만 수집한다.
3. 전체 수집은 데이터베이스 안의 모든 페이지를 수집한다.
4. Source별로 하나의 기본 수집 방식을 설정한다.
5. 복잡한 AND/OR 조건 조합은 후속 버전에서 검토한다.

### 9.5 복습 평가

1. 복습 평가는 4단계다.
2. 다시는 FSRS Again에 매핑한다.
3. 어려움은 FSRS Hard에 매핑한다.
4. 보통은 FSRS Good에 매핑한다.
5. 쉬움은 FSRS Easy에 매핑한다.
6. 평가 완료 시 Review Log를 저장한다.
7. 평가 완료 시 next dueAt을 갱신한다.

### 9.6 변경된 페이지

1. Notion의 수정 시각이 앱에 저장된 마지막 동기화 시각 이후이면 변경된 페이지로 표시할 수 있다.
2. 변경된 페이지는 별도 화면에서 확인할 수 있어야 한다.
3. 변경된 페이지가 감지되어도 앱은 자동으로 dueAt을 변경하지 않는다.
4. 사용자는 변경된 페이지에 대해 "오늘 복습으로 당기기" 또는 "기존 일정 유지"를 선택한다.
5. "오늘 복습으로 당기기"를 선택하면 dueAt을 오늘로 설정한다.
6. "기존 일정 유지"를 선택하면 기존 FSRS 일정과 dueAt을 유지한다.

### 9.7 삭제된 페이지

1. Notion에서 실제 삭제된 페이지는 앱에서도 삭제 또는 삭제 상태로 처리한다.
2. 앱은 삭제된 페이지를 별도 화면에서 확인할 수 있게 한다.
3. 동기화 조회 결과에서 일시적으로 누락된 항목은 즉시 삭제 확정하지 않는다.
4. 권한 오류, 네트워크 오류, API 오류와 실제 삭제는 구분해야 한다.
5. 삭제된 페이지는 active Review Queue와 오늘 복습 목록에서 제외한다.
6. Review Log 보존 여부는 SRS 단계에서 확정한다.

### 9.8 미분류 처리

1. 태그 또는 분류 값이 없는 페이지는 "미분류"로 표시한다.
2. 미분류 항목도 복습 큐에 포함될 수 있다.
3. 미분류 항목은 특정 태그순 보기에서 별도 그룹으로 표시한다.
4. 미분류 여부는 복습 대상 제외 조건이 아니다.

### 9.9 정렬과 랜덤

1. 오늘 복습 목록 기본 정렬은 dueAt 오름차순이다.
2. dueAt이 같은 경우 마지막 복습일이 오래된 항목을 먼저 표시할 수 있다.
3. 랜덤순은 오늘 복습 대상 안에서만 적용한다.
4. 특정 태그순은 태그 또는 분류 속성이 있는 항목을 기준으로 필터링하거나 그룹화한다.
5. 태그 없는 항목은 "미분류" 그룹으로 표시한다.

## 10. 운영 환경 및 제약사항

### 10.1 Electron 데스크톱 앱

본 제품은 1차 범위에서 Electron 기반 데스크톱 앱으로 만든다.

1. 데스크톱 PC에서 실행하는 것을 전제로 한다.
2. Windows 환경을 우선한다.
3. macOS와 Linux 지원은 후속 검토한다.
4. iPad 네이티브 앱은 1차 범위에 포함하지 않는다.
5. iPad 사용은 원격 데스크톱을 통해 PC의 Electron 앱을 조작하는 방식까지 고려한다.

### 10.2 SQLite 저장

앱은 로컬 SQLite에 복습 큐, FSRS 상태, 복습 기록, 동기화 상태를 저장한다.

SQLite에는 다음 데이터가 저장된다.

1. Review Source 설정
2. 필드 매핑
3. Review Item
4. FSRS 상태
5. Review Log
6. Sync Event
7. 삭제/변경 페이지 상태

Notion 페이지 본문 전체는 저장하지 않는다.

### 10.3 Notion API 제약

앱은 Notion API를 사용해 사용자가 지정한 데이터베이스의 페이지를 조회한다.

주의할 점은 다음과 같다.

1. 사용자가 Notion API 토큰을 입력해야 한다.
2. 대상 데이터베이스가 해당 Integration에 공유되어 있어야 한다.
3. Notion 권한이 없으면 페이지를 수집하거나 열람할 수 없다.
4. API 오류 또는 권한 오류는 삭제와 구분해야 한다.
5. Notion 페이지 열람은 로그인 상태와 권한의 영향을 받을 수 있다.

### 10.4 Notion 토큰 저장

MVP에서는 Notion API 토큰 저장 방식이 미확정이다.

후보는 다음과 같다.

1. 로컬 설정 파일 저장
2. OS 보안 저장소 사용
3. 환경 변수 사용

MVP에서는 구현 단순성을 위해 로컬 설정 파일 또는 OS 보안 저장소 중 하나를 선택한다.  
정확한 방식은 SRS/SDD 단계에서 확정한다.

### 10.5 iPad 원격 사용 고려

iPad에서 직접 앱을 실행하는 것은 1차 범위가 아니다.

다만 사용자가 iPad에서 원격 데스크톱으로 PC의 Electron 앱을 조작할 수 있으므로, UI는 다음을 고려한다.

1. 버튼 크기를 너무 작게 만들지 않는다.
2. 드래그 앤 드롭만으로 평가하지 않는다.
3. 난이도 평가 버튼을 항상 제공한다.
4. 문서 열기와 평가 흐름을 단순하게 유지한다.
5. 터치 환경에서 실수로 평가하지 않도록 확인 또는 되돌리기 흐름을 검토한다.

6. 상태
7. 평가 버튼

필요한 기능은 다음과 같다.

1. Source 추가
2. Source 수정
3. Source 삭제
4. Source 활성/비활성 전환
5. Notion Database URL 또는 ID 입력

## 15. 리스크 및 고려사항

### 15.1 Notion 데이터베이스 구조 다양성

데이터베이스마다 속성명이 다르면 앱이 자동으로 제목, 태그, 체크박스 속성을 판단하기 어렵다.

대응 방향은 다음과 같다.

1. Source별 필드 매핑을 제공한다.
2. 필수 매핑과 선택 매핑을 구분한다.
3. 수집 테스트로 매핑 결과를 미리 확인하게 한다.
4. 자동 추론은 후속 편의 기능으로 둔다.

### 15.2 삭제와 동기화 오류 혼동

동기화 결과에서 항목이 사라졌다고 해서 실제 삭제라고 단정할 수 없다.

가능한 원인은 다음과 같다.

1. 페이지 실제 삭제
2. Notion 권한 제거
3. Integration 공유 해제
4. 필터 조건 변경
5. 네트워크 오류
6. API 오류

대응 방향은 다음과 같다.

1. missing, deleted, sync_error 상태를 구분한다.
2. Page ID 재확인 후 삭제 상태로 전환한다.
3. 삭제된 페이지 화면에서 사용자가 최종 정리할 수 있게 한다.

### 15.3 변경된 페이지 처리 기준

Notion 페이지의 수정 시각이 바뀌어도 실제 학습 내용이 의미 있게 바뀐 것은 아닐 수 있다.

대응 방향은 다음과 같다.

1. 변경된 페이지를 자동으로 오늘 복습으로 당기지 않는다.
2. 변경된 페이지 화면에서 사용자가 직접 선택하게 한다.
3. 일괄 처리 기능은 후속 버전에서 검토한다.

### 15.4 Electron 내부 Notion 열람 이슈

Electron 내부 뷰어에서 Notion 페이지가 로그인, 권한, 보안 정책 문제로 열리지 않을 수 있다.

대응 방향은 다음과 같다.

1. 외부 브라우저로 열기 기능을 제공한다.
2. 내부 뷰어 실패 시 명확한 안내를 제공한다.
3. Notion 로그인 상태와 권한 문제를 동기화 오류와 구분한다.

### 15.5 FSRS 상태 저장 오류

FSRS 상태를 단순 dueAt만 저장하면 복습 품질과 통계 확장성이 떨어질 수 있다.

대응 방향은 다음과 같다.

1. fsrsState를 JSON으로 저장한다.
2. Review Log를 별도로 저장한다.
3. 평가 전후 상태를 기록한다.
4. 라이브러리 교체 가능성을 고려해 상태 구조를 SRS/SDD에서 명확히 한다.

### 15.6 랜덤 복습의 재현성

랜덤순 보기에서 매번 순서가 바뀌면 사용자가 혼란스러울 수 있다.

대응 방향은 다음과 같다.

1. 기본은 날짜순으로 한다.
2. 랜덤순은 사용자가 명시적으로 선택할 때만 적용한다.
3. 오늘 세션 내 랜덤 순서 고정 여부는 SRS에서 정의한다.

### 15.7 미분류 항목 누락 위험

태그가 없는 페이지를 필터링 과정에서 제외하면 사용자가 기대한 전체 복습과 다르게 동작할 수 있다.

대응 방향은 다음과 같다.

1. 태그 없는 페이지는 "미분류"로 표시한다.
2. 미분류 항목도 복습 대상 조건을 만족하면 큐에 포함한다.
3. 특정 태그 보기에서도 미분류 그룹 접근 경로를 제공한다.

### 15.8 iPad 원격 사용 UX

iPad에서 원격 데스크톱으로 사용할 경우 드래그 앤 드롭과 작은 버튼이 불편할 수 있다.

대응 방향은 다음과 같다.

1. 버튼 평가를 기본 제공한다.
2. 칸반 드래그는 보조 기능으로 둔다.
3. 클릭 영역을 충분히 크게 만든다.
4. 문서 열람과 평가 흐름을 단순하게 유지한다.

## 16. 개발팀에게

이 제품은 Notion을 대체하는 문서 편집 앱이 아니다.  
또한 Anki처럼 카드 내용을 새로 만드는 앱도 아니다.

1차 목표는 Notion에 이미 쌓인 학습 문서를 **문서 URL 단위로 다시 보게 만드는 복습 큐 앱**이다.

개발 시 다음 원칙을 우선한다.

1. 복습 단위는 Notion Page다.
2. Notion Database는 복습 Source다.
3. 여러 Source는 하나의 Review Queue로 병합한다.
4. Notion Page ID를 중복 판단의 1차 기준으로 삼는다.
5. Notion 페이지 본문 전체는 저장하지 않는다.
6. URL과 Page ID를 저장해 Electron 내부에서 직접 열람한다.
7. Source별 필드 매핑을 제공한다.
8. 수집 기준은 태그/분류, 체크박스, 전체 DB를 모두 지원한다.
9. 복습 평가는 다시, 어려움, 보통, 쉬움 4단계로 제공한다.
10. FSRS 계산 결과로 dueAt을 갱신한다.
11. Review Log를 반드시 저장한다.
12. 삭제와 동기화 오류를 혼동하지 않는다.
13. 변경된 페이지는 자동 처리하지 않고 사용자가 선택하게 한다.
14. 태그가 없는 페이지는 "미분류"로 표시한다.
15. iPad 원격 사용을 고려해 드래그 없이도 평가할 수 있게 한다.
16. MVP는 작게 만들되, 통계와 자동 동기화가 확장될 수 있게 복습 기록을 보존한다.
